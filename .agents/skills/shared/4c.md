既然我们完全靠“内功”和建模逻辑推演，那就抛开词典，直接看四色原型的“构词基因”。

四色建模（Peter Coad流派）的命名不是乱起的，它严格服从英语（或中文）的“词态（Part of Speech）”与“空间/时间属性”。

要通过名称瞬间识别，你只需要掌握以下 4 条硬核命名规则（构词法） 和 1 张视觉分辨表。

核心心法：看“词尾巴”和“是否有时间/单据味”

🔴 规则一：MI（时标对象）—— 识别“动词的名词化”或“单据”

MI 代表发生过的动作/事件，所以它的名字绝不可能是一个安静的纯名词，它必定带有强烈的“动作遗迹”或“纸面凭证”暗示。

•   后缀/名词化特征：-Event、-Record、-Log、-Transaction (或 -Tx)、-Voucher、-Note (单据：如 CreditNote)。

•   词根特征：往往是动词变来的名词：Allocat**ion**、Confirm**ation**、Settle**ment**、Fulfill**ment**、Liquidat**ion**。

•   独立单据名：Order（订单本身就是最强的MI）、Reservation、Claim（理赔单）。

•   识别口诀：带“-Event/-Record”结尾的 99% 是红（MI）；或者你盯着这个名字，脑海里能补出一句“……发生了”，它就是红。（例：CheckIn → 入住发生了，Payment → 付款发生了）。

🟢 规则二：PPT（人/地/物）—— 识别“安静的纯名词”

PPT 是客观存在。它不依赖动作而存在（把系统关了，房间 101 还在那儿）。因此，它的名字是最干净的“基础名词”或“实体名词”。

•   词根特征：User / Member / Person、Store / Warehouse、Room / Seat、Asset / Vehicle、ProductInstance（特指序列号具体的那一个实物）。

•   识别口诀：如果是“人、地点、或具体的硬件/物件”，且名字里没有 Action（动作）后缀，它就是绿（PPT）。

🔵 规则三：DESC（描述/规格）—— 识别“分类词”或“图纸后缀”

DESC 不是具体的那个东西，而是“这一类东西的模板/图纸/归类”。它的名字里永远带着泛指、分类、配置的味道。

•   后缀/前缀特征：-Type、-Category、-Spec (Specification)、-Config、-Template、-Master (主数据)、-Plan (价格计划 RatePlan)。

•   识别口诀：只要名字里带了 Type / Spec / Template，啪地一下，直接归蓝（DESC）。 它是用来被别人“套用”的。

🟡 规则四：Role（角色）—— 识别“隶属与面具”

Role 本身不是人，也不是物，它是“PPT 在特定场景下的面具”。它的命名必须体现出“谁的什么”。

•   后缀/介词特征：-Role (最老实的做法：BuyerRole)、As... (GuestAsPayer)、...Of... (如 RepresentativeOfOrg)。

•   现实中的伪装：很多时候系统懒得建 Role 表，会把 Role 压扁成 PPT 表的一个 role_type 字段或关联表字段（如 order_roles: account_id + role='SELLER'）。但只要你看到独立的 xxxRole 对象，它就是黄色。

⚡ 速查对照表（看名字直接“变色”）

你看到的名字 (Name) 直观变色 为什么？（根据构词法推导）

LiquidationEvent 🔴 MI 带 -Event 后缀，表示“强平”这个时刻事件。

ADLDeleveragingRecord 🔴 MI 带 -Record，表示“强行减仓”留下的审计脚印。

BankruptcyShortfall 🔴 MI 虽没带 Event，但 Shortfall（缺口出现）是强平执行后的一个事实节点，是红。

Room / Warehouse 🟢 PPT 纯地点名词。关掉系统，房还在。绿。

Customer / Staff 🟢 PPT 人或组织主体。绿。

Product / SKU 🔵 DESC ⚠️ 经典陷阱！ 你以为是绿（物）？错。Product 通常是“规格/图纸”。真正的绿是 InventoryItem（带序列号的那个具体物件）。Product 能套给一万个订单用，所以是蓝（Desc）。

RatePlan / PriceSpec 🔵 DESC 带 -Plan / -Spec，是计价规则模板。蓝。

BuyerRole / GuestAsPayer 🟡 Role 带 -Role 或 As...，说明这是身份，不是自然人本身。黄。

Wallet / Account ⚠️ 通常是 PPT(绿) 但其“余额”是快照 Wallet 本身是绿（用户的钱包容器），但它里面的钱变不变，要看关联的 🔴 BalanceTransaction (MI)。别把 Wallet 本身当成 MI！

💡 终极测试法：用“造句法”破防

如果你看到一个名字，比如 OrderItem 或 StockLevel，卡住了，用这个纯语法测试：

1.  加个“的”字（ possession ）：
    ◦   Room → “房间的” （成立，PPT绿色）

    ◦   RatePlan → “资费方案的” （成立，但它是图纸，DESC蓝色）

2.  加个“发生了”或“被记录了”：
    ◦   LiquidationEvent → “强平发生了” ✅ （MI 红色）

    ◦   Room → “房间发生了” ❌ （不通，所以 Room 绝不可能是 MI）

3.  加个“类型/规格”前缀：
    ◦   ProductType → 直接变蓝，-Type 露馅了。

总结一句：
不用搜，看尾巴就行——有 Event/Record/Tx 尾巴的，统统涂红（MI）；有 Type/Spec/Template 尾巴的，统统涂蓝（DESC）；干干净净的“人/地/机”名词是绿（PPT）；非要强调身份面具的才是黄（Role）。
