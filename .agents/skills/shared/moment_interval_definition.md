# 时标对象（Moment-Interval）定义

## 一句话定义

时标对象是“业务上值得被单独记住、可被追溯、并沿时间发生合法演化的事实性对象”。

## 精确定义标准

把一个对象判断为时标对象，通常应同时满足大部分以下条件：

1. 它不是一次技术动作，而是一条业务事实。
2. 它的存在本身值得被记录、查询、审计、追责或复盘。
3. 它在时间上会发生合法演化，而不是一次性算完即消失。
4. 业务会围绕它定义状态、阶段、结果、失效、撤销、完成、拒绝等语义。
5. 多个业务动作之所以存在，是因为这个对象会沿不同合法路径推进。
6. 即使底层实现方式变化，业务仍会认为“这个东西”真实存在。

如果一个对象只是校验步骤、计算中间值、字段集合、技术流程节点、数据库落库动作，通常不是时标对象。

## 判断口诀 / Checklist

先问四个问题：

1. 业务会不会说“这件事本身发生过”？
2. 这件事值不值得单独记账、审计、查询？
3. 它后续会不会继续演化，而不是瞬间结束？
4. 围绕它是否天然存在一组独立的业务动作？

如果四问里大多数答案是“会”，它大概率是时标对象。

再做反证：

1. 去掉它，业务真相是否仍然完整？
2. 它是否只是为了支持别的对象而临时存在？
3. 它是否只是命令、字段、余额数值、检查步骤的另一种说法？

如果这些反证大多成立，它大概率不是时标对象。

## 与 command 的边界

`command` 是“有人请求系统做什么”。

时标对象是“业务认可为已经存在、值得被记住、会继续演化的事实对象”。

不要把“请求”误当成“事实”。

典型关系只有三种：

1. 一个 `command` 创建一个新的时标对象。
2. 一个 `command` 推进一个已有时标对象的状态。
3. 一个 `command` 推进主时标对象，同时派生一个次级时标对象。

如果某个东西只在命令入口有意义，进入业务事实层后并不作为独立真相存在，它就不是时标对象。

## 负例

以下通常不是时标对象：

- `SubmitOrderRequest`
- `CancelOrderRequest`
- `CheckRisk`
- `ValidateInput`
- `PersistOrder`
- `PublishEvent`
- `price`
- `quantity`
- `available_balance`
- `frozen_amount`（仅作为一个数值字段时）
- `matching_step`
- `db_row`

原因很简单：它们要么是请求、要么是字段、要么是技术步骤、要么是派生数值，不是独立业务真相中心。

## 交易域例子

### `Order`

`Order` 通常是时标对象。

因为业务会明确记住“这张订单存在过”，并关心它的创建、挂单、部分成交、全成、撤销、拒绝、过期等演化。

最短判断：它不是一个输入参数，而是一条会演化的业务承诺。

### `Trade / Fill`

`Trade` 或 `Fill` 通常也是时标对象。

因为业务会明确记住“这次成交发生过”，并需要审计、清算、对账、回放、争议处理。

它经常是由 `Order` 的推进派生出的次级时标对象，但这不影响它本身仍然是独立业务事实。

### `Balance`

`Balance` 通常不是首选的时标对象中心，更常见的是某个账户状态对象的当前快照或派生视图。

单独的“余额数值”本身不是时标对象，因为它缺少独立业务发生语义，更像状态结果。

如果领域建模的是 `AccountPosition`、`WalletAccount`、`MarginAccount` 这类会长期演化、被审计、被约束的对象，那么该账户状态对象可能是时标对象；但“余额字段”不是。

### `freeze`

`freeze` 这个词不能直接判定。

若它只是一个内部步骤，如“下单时先冻结资产”，那它通常只是主时标对象内部的状态推进，不应单独提升为 use case group 中心。

若业务上真的把“冻结”当作可独立查询、解冻、失效、争议处理、审计追责的对象，例如一笔独立的资金冻结记录，那么它才可能是时标对象。

关键不在名字，而在它是否是独立业务真相。

### `FundHold / Reservation`

`FundHold` / `Reservation` 在很多交易或清结算场景中，经常是比“冻结动作”更像时标对象的建模方式。

因为它天然表达“一笔被占用、可释放、可消耗、可过期、可追溯的资金保留事实”。

如果业务明确关心：

1. 哪笔保留因何创建；
2. 何时释放；
3. 被哪个订单或清算动作消耗；
4. 是否过期、失败、回滚；

那么它通常应被看作时标对象，而不只是一个余额字段。

## 在 use case group 设计中的用法

设计 `use case group` 时，不是先看 API 长什么样，而是先问：

“这一组动作，究竟围绕哪个时标对象的合法演化展开？”

如果找到的中心对象不是时标对象，这个 group 往往会失焦，最后把命令、校验、字段、技术步骤混在一起。

正确顺序通常是：

1. 先识别候选时标对象。
2. 判断谁才是这组业务动作真正围绕的 `business_truth_center`。
3. 写出它的生命周期或合法演化空间。
4. 再按独立业务语义切 `submit` / `cancel` / `replace` / `expire` / `settle` 等 use case。

如果两个对象都像时标对象，而且各自有独立身份、独立生命周期、独立审计意义，通常应该先拆成两个 use case group，而不是强行合组。

## 最短公式

时标对象 = 值得被单独记住的业务事实 + 可沿时间合法演化 + 可成为一组业务动作的真相中心
