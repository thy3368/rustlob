# 优化延迟

以下优化可能有助于对延迟敏感的交易者：

1. 针对可靠对等节点（如 Hyper Foundation 非验证者）运行非验证节点。
2. 使用 `--disable-output-file-buffering` 运行节点，以便在区块执行后立即获得输出
3. 使用足够的机器规格运行节点，至少 32 个逻辑核心和 500 MB/s 磁盘吞吐量。增加核心可以减少延迟，因为区块执行会更快。
4. 使用节点输出在本地构建订单簿和其他交易所状态，这比 API 提供更快、更细粒度的数据。参见 https://github.com/hyperliquid-dex/order_book_server 了解如何在运行节点的同一台机器上构建订单簿的示例。
5. 节点上的 `--batch-by-block` 将等到区块末尾才写入数据。上面的示例订单簿服务器使用此标志简化逻辑，但进一步优化可以包括关闭该标志并以其他方式推断区块边界。
6. 考虑通过使 nonce 无效来取消待处理订单，而不是垃圾邮件发送取消操作。这将节省用户速率限制，并且如果 nonce 无效交易首先落地，则有保证的成功率。用于 nonce 无效的廉价交易是没有额外字段的 `noop`。
