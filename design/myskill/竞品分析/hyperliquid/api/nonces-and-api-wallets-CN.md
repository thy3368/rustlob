# Nonces 和 API 钱包

## 背景

去中心化 L1 必须防止重放攻击。当用户签署 USDC 转账交易时，接收者不能多次广播它以耗尽发送者的钱包。为解决此问题，以太坊为每个地址存储一个 "nonce"，这是一个从 0 开始的数字。每笔交易必须使用恰好 "nonce + 1" 才能被包含。

## API 钱包

这些在文档中也称为 `agent wallets`。主账户可以批准 API 钱包代表主账户或任何子账户签名。

注意 API 钱包仅用于签名。要查询与主账户或子账户关联的账户数据，您必须传递该账户的实际地址。一个常见的陷阱是使用代理钱包，导致空结果。

## API 钱包修剪

API 钱包及其关联的 nonce 状态可能在以下情况下被修剪：

1. 钱包被注销。当发送 ApproveAgent 操作注册新的未命名 API 钱包时，现有的未命名 API 钱包会发生这种情况。当发送 ApproveAgent 操作且名称匹配时，现有的命名 API 钱包也会发生这种情况。
2. 钱包过期。
3. 注册代理的账户不再有余额。

**重要：** 对于以编程方式使用 API 钱包的人，"强烈建议不要重复使用其地址"。一旦代理被注销，其使用的 nonce 状态可能会被修剪。在未来使用时生成新的代理钱包以避免意外行为。例如，一旦 nonce 集被修剪，先前签名的操作可以被重放。

## Hyperliquid Nonces

以太坊的设计不适用于链上订单簿。做市策略每秒可以发送数千个订单和取消。要求在区块链上精确排序包含将破坏任何策略。

在 Hyperliquid 上，"每个地址存储 100 个最高 nonce"。每笔新交易必须有比此集合中最小 nonce 更大的 nonce，并且以前从未使用过。Nonce 按签名者跟踪，即如果使用地址的私钥签名则为用户地址，如果使用 API 钱包签名则为代理地址。

Nonces 必须在 `(T - 2 天, T + 1 天)` 内，其中 `T` 是交易区块的 unix 毫秒时间戳。

以下步骤可能有助于从中心化交易所移植自动化策略：

1. 每个交易进程使用一个 API 钱包。注意 nonces 按签名者存储（即私钥），因此由相同 API 钱包签名的不同子账户将共享 API 钱包的 nonce 跟踪器。建议为不同子账户使用单独的 API 钱包。
2. 在每个交易进程中，有一个任务每 0.1 秒定期批量处理订单和取消请求。建议将 IOC 和 GTC 订单与 ALO 订单分开批量，因为仅 ALO 订单的批次被验证者优先考虑。
3. 交易逻辑任务将订单和取消发送到批处理任务。
4. 对于每批订单或取消，获取并递增原子计数器，确保地址的唯一 nonce。如有需要，原子计数器可以快进到当前 unix 毫秒。

这种结构对于 2 秒内的无序交易具有鲁棒性，这对于地理位置靠近 API 服务器的自动化策略应该足够。

## 子账户和金库用户建议

注意 nonces 按签名者存储，即用于签名交易的私钥地址。因此，建议每个交易进程或前端会话使用单独的私钥签名。特别是，用于用户、金库或子账户的单个 API 钱包都共享相同的 nonce 集。

如果用户想并行使用多个子账户，在主账户下生成两个单独的 API 钱包，并为每个子账户使用一个 API 钱包会更容易。这避免了每个子账户使用的 nonce 集之间的冲突。
